# Недостатки модели (сводно)

## 1. Ошибки направления метрик (прямая/обратная) и РАНГ-метрик

> **Статус: в основном устранён** (`5ae2412`). Перепрогон: M8 исправлен —
> рекомендации теперь с верным направлением. Остаток — формулировки РАНГ-метрик
> в E3.

Самая частая и тяжёлая категория. Модель не выдерживает направление для «обратных»
метрик и особенно для РАНГ-метрик (`metric_type=обратная`, меньше = лучше).

- **test_metrics / M8** — называет худший в секторе показатель «успешным опытом»:
  > «Кузнецова Татьяна Андреевна показывает значительное превышение плана по доле
  > переводов… проанализировать **успешный опыт** таких сотрудников и внедрить»

  У Кузнецовой доля переводов 29.96 — это `плохо` (обратная метрика, антилидер сектора).
- **sample_good / E3, sample_weak / E3, sample_stable / E3** — РАНГ-метрики:
  > «Ранг AHT, Ранг доли переводов и Ранг производительности находятся **выше
  > запланированных значений, что свидетельствует о хорошем положении**»

  Для РАНГ-метрики (обратная) «выше плана» = хуже. Снимок: `РАНГ Производительность`,
  `РАНГ AHT` на последней неделе — `хуже_плана`, а модель относит их к выполненным.
- **test_metrics / M8** — то же: «РАНГ Производительность значительно выше плана».

## 2. Подмена смысла вопроса: «просела» (тренд) ≠ «хуже плана» (отклонение)

> **Статус: устранён** (`5ae2412`). Перепрогон: M5 и A5 теперь выбирают метрику
> по тренду, а не по отклонению от плана.

На вопрос «какая метрика **просела** сильнее всего» модель отвечает метрикой с
наибольшим **отклонением от плана**, а не с наибольшим падением динамики.

- **test_metrics / M5** — ответ «Доля переводов»; эталон (`find_flags('trend')`):
  сильнее всего просела `Влияние тематик на Talk&Hold` (Попов Роман Максимович,
  wow −1443 %).
- **test_metrics_agile / A5** — ответ «Bug Rate (+350 % от плана)»; эталон:
  `Влияние компонентов на Cycle Time` (Беляев Илья, wow −1400 %).

## 3. Галлюцинации и прямые расхождения с данными

- **test_metrics / M4** — «самая сильная аномалия у Новиковой Юлии Никитны…
  z-score 2.97». Эталон `find_flags('anomaly')`: топ-аномалия — **Кузнецов Роман
  Алексеевич**, Доля переводов / Ликвидность, z=2.39. Значения z=2.97 в данных нет;
  вторая «аномалия» — «Соколова Мария / HOLD TIME z=2.84» — у Соколовой аномалия по
  `Влияние тематик` (z=−2.17), не по HOLD TIME.
- **test_metrics / M5** — утверждает: «метрика Влияние тематик на Talk&Hold **вообще
  отсутствует** в выборке данных». Метрика есть в датасете.
- **test_metrics / M1** — аномалии приписаны не тем людям (Новикова Юлия, «Соколова /
  HOLD TIME» — нет в `find_flags('anomaly')`); значение 29.96 % приписано сразу двум
  сотрудникам (реально — только Кузнецовой Татьяне).
- **test_metrics_agile / A7** — «Беляев Александр и Новиков Максим показывают
  **худшие** результаты по Code Quality». Снимок: они rank 1 и rank 2 — **лучшие**;
  худшие — Козлов Александр (66.4) и Беляев Илья (72.8).
- **sample_weak / E6** — упоминает несуществующую метрику «АСН (Average Speed of
  Answer)»; такой метрики в датасете нет.

## 4. Игнорирование актуальной даты — устаревшие значения как «сейчас»

На вопрос «в чём **сейчас** главная проблема» модель приводит значения прошлых недель.

- **sample_good / E1** — «по банковским счетам доля переводов 29.83 %». На последней
  неделе (2026-05-11) у этого элемента 7.94 (лучше плана). 29.83 — из ранней недели.
- **sample_declining / E1** — «Ликвидность 29.8 %, Финмониторинг 29.33 %…». Последняя
  неделя: максимум по below_plan — 20.75. Приведены устаревшие пиковые значения.
- **sample_rising / E1** — для вопроса «сейчас» собрал данные за `2026-05-04`, хотя
  последняя неделя — `2026-05-11`.

## 5. Внутренние противоречия в одном ответе

- **sample_rising / E3** — «Производительность» одновременно в списке «выполняет план»
  и «не выполняет план».
- **sample_good / E4, sample_declining / E4** — называет один компонент «дальше всего
  от бенчмарка», а рядом приводит другой с бо́льшим процентом отклонения
  (good: HOLD +71.89 % назван худшим при AUX +83.81 %; declining: ACD +31 % назван
  худшим при ACW +41 %).
- **sample_good / E6, sample_declining / E1, sample_stable / E6** — в одной фразе
  «находится значительно **ниже плана**» и «фактическое значение **превышает план**».
- **test_metrics / M4** — «резко **вырос** на 9.71 % за неделю (**падение**)».

## 6. Мусорные аргументы инструментов

GigaChat подставляет в аргументы инструментов мусор и несуществующие значения
(подтверждает ранее зафиксированное поведение модели).

- **sample_stable / E2** — критично: вызовы `find_flags(metric='\n}')`,
  `compare(metric=',\n    ')` (и без обязательного `person`). Данные по
  производительности не собраны — ответ целиком про другую метрику
  («Влияние тематик»), на вопрос «растёт или падает» фактически не отвечено.
- **sample_good / E3, sample_stable / E3** — `find_flags(metric='*')` (заглушка `*`).
- **sample_declining / E4** — `find_flags(kind='benchmark')` — выдуман несуществующий
  вид флага.
- **test_metrics / M6** — `get_metric(person='Сектор 1')` — несуществующий человек.

## 7. Неполнота: обрезка топ-25 и упущенные данные

- **test_metrics_agile / A2** — на «кто не выполняет план» названы 4 человека и только
  баг-метрики. Эталон: невыполнение плана есть ещё у Соколова Дмитрия и Беляева
  Александра и по метрикам Velocity, Code Quality, Cycle Time, Sprint Commitment
  Accuracy, РАНГ-метрикам. `find_flags('below_plan')` отдаёт топ-25 строк,
  отсортированных по величине отклонения; крупные отклонения Bug Rate вытеснили
  остальные метрики, и модель их «не увидела».

## 8. Неверная интерпретация технических артефактов выдачи

- **sample_declining / E1, test_metrics / M5** — «всего выявлено **25 случаев**».
  25 — это лимит строк выдачи `find_flags`, а не число случаев; модель приняла
  ограничение пагинации за факт.
- **sample_good / E5, sample_declining / E5** — «занимает **третье место** в рейтинге».
  3 — это значение метрики `РАНГ Доля переводов` (fact=3.0), а не позиция среди
  коллег; в одиночном датасете коллег для рейтинга нет.

## 9. Выход за лимит шагов сбора и переисследование

> **Статус: устранён** (`5ae2412`). Мягкий бюджет в 18 вызовов даёт штатную
> раннюю остановку: все 7 ячеек при перепрогоне завершились `completed=true`.

- 7 ответов из 51 — `completed=false` (упор в лимит рекурсии 50/25 вызовов):
  `sample_rising` E3/E6, `sample_declining` E3/E6, `agile` A5/A6/A7.
- Патология переисследования: `agile / A5` и `A6` вызвали `describe_metric`
  практически для всех 25 метрик подряд.
- Даже среди `completed=true` много ответов с 14–25 вызовами там, где хватило бы
  2–4 (напр. `sample_good / E1` — 25 вызовов на «в чём главная проблема»).

## 10. Прочее

- **sample_declining / E2** — на «растёт или падает» для явно деградирующего оператора
  (Производительность 25.44→12.05) ответ начинается с «стабильный **рост**»; падение
  упомянуто вскользь ниже. Неверный главный вывод.
- **Несогласованность между вопросами**: `sample_good / E1` называет «Долю переводов»
  главной проблемой, а `E5` на том же датасете — «хорошей» (E1 берёт худшие
  по-элементные значения, E5 — агрегат). Пользователь получит противоречивые ответы.

# Что работает хорошо

- **Числовая достоверность высокая.** Из 51 ответа лишь одно число не нашло
  подтверждения в транскрипте (`sample_good / E7`, «+39.3 %»). Модель почти не
  выдумывает числа — берёт их из инструментов; ошибки сосредоточены в
  *интерпретации*, а не в фабрикации данных.
- **Ловушки отрабатываются корректно.** Запрос peer-сравнения при одном операторе
  (E7 на всех 5 сэмплах) — всюду честный ответ «сравнение невозможно, peer_count=1».
  Вопрос про несуществующую метрику (A8, «выручка») — честный отказ без единого
  вызова инструмента.
- **Простые детерминированные запросы точны.** M2 (15/15 не выполнивших план),
  M3 и A3 (топ-рейтинги — точное совпадение с `rank()`), A4 (топ-аномалия) —
  безошибочно, когда задача сводится к одному корректному вызову инструмента.
- **Прямой вопрос о направлении** («хорошая ли доля переводов», E5) — направление
  обратной метрики выдержано на всех 5 сэмплах; M7 («лучшая доля переводов») —
  корректно выбран минимум.

# Краткие рекомендации

1. **Направление метрик** — усилить в промпте синтеза (стадия 2) и в выдаче
   инструментов жёсткую привязку вердикта к `metric_type`; для РАНГ-метрик
   («обратная») «выше плана» всегда хуже. Это источник самых грубых ошибок (кат. 1).
   — ✅ Сделано (`5ae2412`).
2. **Различать «тренд» и «отклонение от плана»** — в описаниях `find_flags`
   (`kind='trend'` vs `below_plan`) и в промпте явно развести «просела» (динамика)
   и «хуже плана» (отклонение). Кат. 2. — ✅ Сделано (`5ae2412`).
3. **Скоуп по дате** — для вопросов «сейчас/сегодня» принудительно подставлять
   последнюю неделю; сейчас агент тянет исторические пики. Кат. 4.
4. **Артефакты выдачи** — не показывать модели «сырой» лимит в 25 строк как данные;
   к усечённой выдаче `find_flags` добавлять явную пометку «показаны N из M», а ещё
   лучше — отдавать агрегаты, а не топ-N. Кат. 7, 8.
5. **Лимит шагов** — 7/51 прогонов не уложились; пресекать `describe_metric` по всем
   метрикам подряд (кат. 9), например кэшировать описания или ограничивать их число.
   — ✅ Сделано: мягкий бюджет вызовов инструментов (`5ae2412`).
6. **Мусорные аргументы** — расширить валидацию (как у `_unknown_metric`) на все
   текстовые аргументы инструментов: ловить `'*'`, переводы строк, несуществующих
   людей и виды флагов до выполнения запроса. Кат. 6.
