# Отчёт: польза для непонимающего руководителя

## Параметры

- Чат-провайдер: **GigaChat-2-Max**. Прогон: `eval_value.py`, 42 исхода
  (7 датасетов × 6 ситуаций), 12.5 мин. Сырые данные — `eval_value_out/`.
- Идея теста: за каждым тестом «сидит руководитель, который ничего не понимает» —
  не знает метрик, не знает, что спросить. Проверяется не фактическая
  корректность (это делает `eval_run.py`), а **получает ли он реальную пользу**.
- Оценку полезности (качественную) делал ассистент по дампам — не GigaChat-судья.

## Что проверялось — 6 ситуаций растерянного руководителя

| ID | Ситуация | Вход |
|---|---|---|
| S1 | Холодный старт | открыл инструмент, ничего не спросил → стартовый экран |
| S2 | Размытый вопрос | «Как дела, всё нормально?» |
| S3 | «Что мне делать» | «На что обратить внимание и что предпринять?» |
| S4 | Непонятный термин | «Что значит „{метрика}“ простыми словами и почему важно?» |
| S5 | Запрос вне возможностей | «Дай прогноз на следующий месяц» |
| S6 | Переход по подсказке | первый предметный вопрос из стартовых подсказок |

**Рубрика пользы:** понятность (простой язык, без жаргона) · приоритизация
(что главное) · действенность (есть конкретный шаг) · навигация (что спросить
дальше) · честность (без выдумок).

## Сводка

Вердикт: **✓** руководитель получает пользу · **~** частично (польза есть, но
с заметным изъяном) · **✗** пользы нет.

| Датасет | S1 | S2 | S3 | S4 | S5 | S6 |
|---|---|---|---|---|---|---|
| sample_good | ~ | ~ | ✓ | ✓ | ~ | ✓ |
| sample_weak | ✓ | ~ | ✓ | ✓ | ~ | ✓ |
| sample_rising | ✓ | ~ | ✓ | ✓ | ~ | ✓ |
| sample_declining | ~ | ~ | ✓ | ✓ | ~ | ✓ |
| sample_stable | ✓ | ~ | ✓ | ✓ | ~ | ✓ |
| test_metrics | ✓ | ~ | ✓ | ✓ | ~ | ✓ |
| test_metrics_agile | ✓ | ~ | ✓ | ✓ | ~ | ✓ |

Итог по ситуациям: **S3, S4, S6 — польза стабильно есть** (✓ во всех 7);
**S1 — есть** (5/7, 2 с изъяном); **S2 и S5 — только частично** (0/7 на ✓).
Ни одна ситуация не оказалась полностью бесполезной — но в двух из шести
растерянный руководитель пользу недополучает.

## По ситуациям

### S1 — Холодный старт · ✓ (изъян в 2/7)

Стартовый экран сразу даёт суть: абзац «ГЛАВНОЕ» простым языком называет
1–2 ключевые проблемы, ниже — находки и 5 готовых вопросов «С чего начать».
Для не знающего, что спросить, — это рабочая точка входа.

Изъян: в `sample_good` и `sample_declining` абзац «ГЛАВНОЕ» пишет про «Долю
переводов» «значительно **ниже** плана» / «**низкую** долю переводов», хотя
метрика обратная и факт ВЫШЕ плана (= хуже). Остаток недостатка №1 — правки
направления не дошли до промпта брифа (`_BRIEF_PROMPT`).

### S2 — Размытый вопрос «как дела» · ~ (все 7)

Структура хорошая: агент даёт приоритизированную оценку и заканчивает
рекомендацией. **Но понятность проваливается** — ответ написан как BI-сводка:
`AHT`, `HOLD TIME`, `ACD TIME`, `ACW TIME`, `AUX PEREZVON TIME`, `Adherence`,
`РАНГ` — аббревиатуры сыплются без единой расшифровки. Руководитель, который
«ничего не понимает», спросил «всё нормально?» и получил стену жаргона.
Он уловит общий тон («есть проблемы»), но не сами проблемы.

Дополнительно — остаток №1: `sample_stable / S2` пишет «AHT находится
значительно **ниже** запланированного уровня (425.0 против 320.0)» — 425 > 320,
это выше; вердикт «плохо» верный, формулировка нет.

### S3 — «Что мне делать» · ✓ (все 7)

Самая сильная ситуация. Агент даёт приоритизированные проблемы + нумерованные
рекомендации, часто с конкретными действиями (коучинг, перераспределение
задач, ретроспективы) и именами сотрудников. Именно то, что нужно
руководителю. Жаргон есть, но рекомендации конкретны и действенны.

Мелкий изъян: на одиночных датасетах (`sample_rising`, `sample_stable`) ответ
иногда обращается ко второму лицу — к самому оператору («ваша
производительность», «обсудите с руководством»), хотя вопрос задавал
руководитель ПРО оператора.

### S4 — Непонятный термин · ✓ (все 7)

Образцово. На «что значит „{метрика}“ простыми словами» агент даёт ясное
определение без жаргона + «почему вам как руководителю это важно». Один вызов
инструмента, быстро. Это и есть та понятность, которой не хватает в S2.

### S5 — Запрос вне возможностей · ~ (все 7)

Честно: на «дай прогноз» агент не выдумывает, а сообщает, что прогнозирования
в данных нет (честность — отлично). **Но нет разворота**: лишь часть ответов
вяло предлагает «уточните запрос», и ни один не говорит «зато я могу показать
вам X, Y». Растерянный руководитель упёрся в «нет» и остался без выхода.

### S6 — Переход по подсказке · ✓ (все 7)

Предметный вопрос из стартовых подсказок ведёт к содержательному ответу:
величина отклонения, направление метрики, причина, рекомендация. Связка
«не знаю, что спросить → подсказка → полезный ответ» работает.

## Что улучшить (для отдельной задачи)

1. **S2 — расшифровывать жаргон.** В промпт синтеза: при размытых/обзорных
   вопросах непонятную аббревиатуру метрики при первом упоминании пояснять
   простыми словами (как это уже делает S4). Главный провал понятности.
2. **S5 — разворот при отказе.** В промпт: отказывая в невозможном, предлагать
   1–2 вопроса, на которые агент реально ответит.
3. **Бриф — направление метрик.** Правила направления (недостаток №1) не дошли
   до `_BRIEF_PROMPT`; абзац «ГЛАВНОЕ» иногда инвертирует «доля переводов».
4. **S3 — адресат.** На одиночных датасетах вопрос руководителя про оператора
   агент иногда отвечает оператору; держать третье лицо.

## Замечание о харнессе

В дампах `eval_value_out/` блок «На что посмотреть» показывает сырые ярлыки
(`[below_plan]`, `[trend]` …) — это упрощение рендера `eval_value.py`. Реальный
CLI (`analyzer/cli.py`) выводит русские ярлыки («хуже плана», «спад» …), поэтому
к оценке S1 это не относится.
