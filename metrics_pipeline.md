# Как обрабатываются метрики

### json_analyzer · пайплайн

От сырого JSON — к готовым выводам, по которым отвечает агент-аналитик.

---

## Обзор пайплайна

```
Входной JSON
   │  разворачивание дерева
   ▼
Плоские строки-метрики      → таблица metrics
   │  производная аналитика
   ▼
Готовые выводы              → таблица metric_analytics
   │  эмбеддинги
   ▼
Семантический кэш (pgvector)
   │  стартовый бриф
   ▼
Агент отвечает пользователю
```

Пять стадий. Первые три — детерминированные, без участия LLM.

---

## Шаг 1. Входной JSON

Форма входа фиксирована для любого домена:

```
{ "me": <человек>, "employees": [ <человек>, ... ] }
```

- `me` — это руководитель, для которого работает агент.
- `employees` — его сотрудники.
- У каждого человека — **дерево метрик**: метрика может содержать
  вложенные `child_metrics`, те — свои, и так вглубь.

Колл-центр в примерах — лишь один домен. Разработчики, менеджеры,
руководители — та же структура, другие названия метрик.

---

## Объект «человек» и объект «метрика»

**Человек** — ФИО, должность, подразделение, роль (я / сотрудник).

**Метрика** — один показатель за одну неделю:

- `fact` — фактическое значение, `plan` — план, `benchmark` — ориентир.
- `metric_type` — **прямая** (чем больше, тем лучше) или
  **обратная** (чем меньше, тем лучше). Задаёт направление «хорошо».
- `element` — разрез (продукт, сегмент); `null` — это агрегат по всем.
- `date` — неделя, к которой относятся числа.

---

## Шаг 2. Разворачивание дерева

Загрузчик рекурсивно обходит дерево метрик каждого человека.

- **1 узел дерева → 1 плоская строка.** Иерархия не теряется:
  у строки есть `depth` (уровень) и `parent_uid` (ссылка на родителя).
- Каждой строке даётся `metric_uid` — сквозной уникальный ключ.
  Исходный `id` для этого не годится: он повторяется на каждом
  разрезе `element`.

Результат — единый плоский список строк вместо вложенных деревьев.

---

## Объект: таблица `metrics`

Плоский список загружается в таблицу `metrics` встроенной базы
(SQLite в оперативной памяти).

- Одна строка = одна метрика одного человека за одну неделю
  на одном разрезе.
- Хранит **только сырые данные** из JSON — ничего не досчитано.
- База живёт в памяти и пересобирается заново при каждом запуске.

Это «сырой слой» — основа для следующего шага.

---

## Шаг 3. Производная аналитика

Отдельный модуль детерминированно (без LLM, по формулам) досчитывает
выводы по каждой строке и кладёт их в **таблицу-спутник**
`metric_analytics`.

Считаются четыре группы производных:

1. Отклонения от плана и бенчмарка
2. Динамика и тренд
3. Сравнение с коллегами
4. z-score и аномалии

Дальше — по каждой группе.

---

## Отклонение от плана и бенчмарка

Поля: `plan_dev_abs`, `plan_dev_pct`, `plan_status`
(и так же `benchmark_dev_abs/pct`, `benchmark_status`).

**Алгоритм** — для каждой строки, факт против плана:

1. Нет `fact` или `plan` → все три поля пустые.
2. `plan_dev_abs = fact − plan`.
3. `plan_dev_pct = plan_dev_abs / plan × 100` (если `plan = 0` —
   процент пустой).
4. Статус:
   - отклонение нулевое или `|plan_dev_pct| < 1%` → **в плане**;
   - иначе сверяем знак отклонения с направлением метрики:
     прямая + факт выше **или** обратная + факт ниже → **лучше плана**;
     иначе → **хуже плана**.

Против `benchmark` — тот же алгоритм, метки «лучше / на уровне /
хуже бенчмарка».

---

## Динамика: изменение неделя-к-неделе

Поля: `wow_change_abs`, `wow_change_pct`.

**Алгоритм:**

1. Строки группируются в серии по ключу
   *(человек, метрика, разрез)*.
2. Серия сортируется по дате.
3. Для каждой недели берётся предыдущая в серии:
   - `wow_change_abs = fact − факт_прошлой_недели`;
   - `wow_change_pct = wow_change_abs / факт_прошлой_недели × 100`
     (прошлый факт = 0 → процент пустой).
4. Если у текущей или прошлой недели факт пустой — изменение не
   считается.
5. У первой недели серии изменения нет — поля пустые.

---

## Динамика: тренд серии

Поле: `trend` — одно значение на всю серию.

**Алгоритм:**

1. Берём серию *(человек, метрика, разрез)*, сортируем по дате,
   отбрасываем пустые факты.
2. Меньше 2 значений → тренд пустой.
3. `first` — первое значение, `last` — последнее.
4. `change = (last − first) / |first| × 100`
   (если `first = 0`: 0% при `last = 0`, иначе 100%).
5. `|change| < 5%` → **стабильно**; иначе **рост** (`last > first`)
   или **падение**.
6. Полученный тренд проставляется всем строкам серии.

Тренд — движение *числа*, без учёта направления: «рост» обратной
метрики на деле ухудшение. Это различает уже агент.

---

## Сравнение с коллегами: группа и статистика

Поля: `peer_mean`, `peer_std`, `peer_count`.

**Группа сравнения** — строки с одинаковыми
*(метрика, разрез, неделя)*. Руководитель (`me`) в группу не входит.

**Алгоритм:**

1. В группе берём строки с непустым фактом → их число `peer_count`.
2. `peer_count < 2` → группа пропускается, все peer-поля пустые.
3. `peer_mean` — среднее арифметическое фактов группы.
4. `peer_std` — стандартное отклонение по группе
   (популяционное, делитель `n`).

---

## Сравнение с коллегами: ранг и процентиль

Поля: `peer_rank`, `peer_percentile`.

**Алгоритм** (в той же группе сравнения):

1. Сортируем сотрудников по факту с учётом направления метрики:
   прямая — по убыванию, обратная — по возрастанию (лучший первым).
2. `peer_rank` — позиция в этом порядке, с 1 (1 = лучший).
3. `peer_percentile = (1 − idx / (n − 1)) × 100`,
   где `idx = peer_rank − 1`, `n = peer_count`.
   Лучший → 100, худший → 0, остальные — линейно между.

---

## z-score и аномалии

Поля: `zscore`, `is_anomaly`.

**Алгоритм** (в той же группе сравнения):

1. Для каждой строки `zscore = (fact − peer_mean) / peer_std`.
   Если `peer_std = 0` (все значения равны) → `zscore = 0`.
2. z-score = на сколько стандартных отклонений факт отстоит от
   среднего группы. Знак не зависит от направления метрики:
   у обратной метрики положительный z-score — это хуже.
3. `is_anomaly = 1`, если `|zscore| ≥ порог`.
   Порог по умолчанию **2.0** (`ANOMALY_ZSCORE_THRESHOLD`).
4. Нужно ≥ 2 человек в группе — иначе peer-поля пустые и аномалия
   не считается.

---

## Объект: таблица `metric_analytics`

Результат шага 3 — таблица `metric_analytics`: по строке на каждую
метрику, 16 производных полей (отклонения, динамика, тренд,
peer-статистика, z-score, флаг аномалии).

- Связана с `metrics` через `metric_uid`.
- Все числовые значения при записи округляются до 4 знаков.
- **Именно её агент читает** через свои инструменты — он не считает
  числа на лету, а берёт готовые выводы.

---

## Шаг 4. Семантический кэш (эмбеддинги)

Чтобы находить метрики по смыслу запроса, а не по точному названию,
их тексты переводятся в векторы.

- Кодируются названия метрик, описания и значения `element`.
- Векторы хранятся в PostgreSQL с расширением pgvector.
- Кэш **накапливается** между запусками и доменами: каждый текст
  кодируется один раз.

---

## Шаг 5. Стартовый бриф

Чтобы пользователь сразу понимал, что важно и о чём спросить, при
старте по `metric_analytics` собираются:

- **Находки** — топ-проблемы и сильные стороны: хуже плана, спад
  динамики, аномалия, лучше плана.
- **Наводящие вопросы** — сформированы из этих же находок, поэтому
  каждый ведёт на уже посчитанный инсайт.

Это первый экран: холодный старт без «не знаю, что спросить».

---

## Итог

**Было:** дерево вложенных чисел в JSON, без выводов.

**Стало:**

- плоские строки фактов в таблице `metrics`;
- готовые выводы в таблице `metric_analytics`;
- семантический индекс метрик;
- стартовый бриф с находками и вопросами.

Агент отвечает по **производным метрикам**, а не считает на лету —
быстро, воспроизводимо и одинаково для любого домена.
