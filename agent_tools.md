# Инструменты агента

### json_analyzer · tools

Как агент-аналитик добывает данные: 11 типизированных инструментов
вместо прямого доступа к базе.

---

## Принцип: только инструменты

Агент **не имеет прямого доступа к базе** и **не пишет SQL**.

- Все данные — через 11 типизированных инструментов.
- У каждого строгая сигнатура (имя + типы параметров); внутри он сам
  выполняет параметрический запрос к SQLite или семантический поиск
  в pgvector.
- «Сырого SQL» нет — модель не формирует произвольный запрос, а лишь
  вызывает готовый инструмент с аргументами.

Это убирает целый класс ошибок и инъекций.

---

## Агент работает в две стадии

```
Вопрос пользователя
        │
        ▼
┌─ Стадия 1 — СБОР ──────────────┐
│  цикл вызовов инструментов     │
│  (11 инструментов подключены)  │
└────────────┬───────────────────┘
             │ собранные данные
             ▼
┌─ Стадия 2 — СИНТЕЗ ────────────┐
│  один вызов модели             │
│  (инструментов нет)            │
└────────────┬───────────────────┘
             ▼
      Ответ пользователю
```

Инструменты работают только на стадии 1 — сборе фактов.

---

## 11 инструментов — четыре роли

**Навигация по датасету** — что вообще есть:
`schema_overview` · `list_people` · `describe_metric`

**Разрешение неточных формулировок:**
`resolve_entity`

**Получение значений метрик:**
`get_metric` · `compare` · `rank` · `aggregate` · `metric_tree`

**Готовые выводы аналитики:**
`find_flags` · `analytics_summary`

---

## schema_overview — обзор датасета

Полный состав загруженных данных одним вызовом.

**Параметры:** нет.

**Возвращает:** список метрик с типами и единицами, значения
`element` (продукты/разрезы), людей, диапазон дат.

**Когда:** в начале — чтобы узнать точные названия для аргументов
других инструментов.

---

## list_people — люди в датасете

Список людей с фильтрами.

**Параметры:**
- `role` — `me` (руководитель) или `employee`;
- `post` — должность, `depart` — подразделение;
- `name_query` — подстрока ФИО для поиска.

**Когда:** найти сотрудника, узнать состав команды, проверить ФИО.

---

## describe_metric — смысл метрики

Описание одной метрики и её «паспорт».

**Параметры:** `metric` — точное название.

**Возвращает:** описание, тип (`прямая` — больше лучше /
`обратная` — меньше лучше), единицу измерения, период расчёта.

**Когда:** перед интерпретацией значений — направление метрики
критично. Не вызывается «для каждой метрики подряд».

---

## resolve_entity — неточное → точное

Превращает описательную формулировку в каноничное имя сущности.

**Параметры:**
- `text` — как пользователь назвал сущность;
- `kind` — `metric` / `element` / `person`.

**Как работает:**
- `metric` / `element` — семантический поиск: `text` кодируется в
  вектор, ищется ближайшее в pgvector-кэше эмбеддингов;
- `person` — поиск по подстроке ФИО.

**Когда:** метрика/продукт/человек названы неточно или описательно.

---

## get_metric — значения метрики

Значения `fact`/`plan`/`benchmark` плюс производная аналитика:
статусы отклонений, динамика, тренд, peer-ранг, флаг аномалии.

**Параметры:**
- `metric` — точное название (обязателен);
- `person` — ФИО / часть ФИО / табельный номер (опц.);
- `element` — продукт/разрез; не указан → агрегат по метрике;
- `date` — неделя `YYYY-MM-DD` (опц.).

**Когда:** нужны конкретные значения и аналитика по одной метрике.

---

## compare — динамика по неделям

Значения метрики во времени для **одного** человека —
с полями `wow_change_pct` и `trend`.

**Параметры:**
- `metric` — точное название (обязателен);
- `person` — **обязателен**: динамика всегда по человеку;
- `element` — не указан → агрегат;
- `dates` — список недель для среза (опц.).

**Когда:** «как менялось у X». Найти, у кого сильнее спад/рост по
всем сотрудникам, — это `find_flags(kind='trend')`, не `compare`.

---

## rank — рейтинг сотрудников

Сотрудники, упорядоченные по метрике на конкретную неделю.

**Параметры:**
- `metric` — точное название (обязателен);
- `date` — неделя (обязателен);
- `element` — не указан → агрегат;
- `post` — фильтр по должности.

**Когда:** «кто лучший/худший по метрике». Направление уже учтено:
`peer_rank = 1` — лучший.

---

## aggregate — сводка по группам

Агрегаты значений метрики: среднее, минимум, максимум, сумма,
количество.

**Параметры:**
- `metric` — точное название (обязателен);
- `group_by` — `person` / `element` / `date` / `post`;
- `date`, `element` — опциональные фильтры.

**Когда:** «среднее по продуктам», «сумма по должностям» — итоги
в разрезе группы.

---

## metric_tree — состав метрики

Метрика со всеми дочерними `child_metrics` **и** аналитикой по
каждому узлу — один вызов вместо обхода компонентов.

**Параметры:**
- `metric` — метрика-корень (без неё → метрики верхнего уровня);
- `person`, `date` — рекомендуется задавать (иначе строк много).

**Возвращает:** дерево узлов с отклонениями от плана/бенчмарка,
трендом, `influent_percent`.

**Когда:** «почему» и «из чего состоит» — какая часть метрики просела.

---

## find_flags — готовые проблемы и находки

Предрассчитанные заметные строки, отсортированные по силе:
первая строка — самая значимая.

**Параметры:**
- `kind` — вид флага (обязателен);
- `date`, `metric`, `element` — опциональные фильтры.

**Виды флага:**
- `below_plan` — факт хуже плана (проблемные места);
- `above_plan` — факт лучше плана (сильные стороны);
- `anomaly` — статистический выброс (|z-score| выше порога);
- `trend` — динамика: кто сильнее просел/вырос за недели.

«Хуже плана» (отклонение) и «просела динамика» (тренд) — разные
вопросы: выбирай `kind` строго по смыслу.

---

## analytics_summary — стартовая сводка

Детерминированная сводка по всему датасету.

**Параметры:** нет.

**Возвращает:** охват датасета, средние по ключевым метрикам на
последней неделе, топ аномалий, счётчики трендов.

**Когда:** общий «снимок» состояния без перебора метрик по одной.

---

## Защита цикла сбора

Стадия 1 — это цикл; у него три ограничителя:

- **Лимит выдачи** — слишком длинный ответ инструмента обрезается
  (строки идут по убыванию значимости — нужное остаётся).
- **Защита от повторов** — повторный вызов с теми же аргументами
  не выполняется: возвращается подсказка завершить сбор.
- **Бюджет вызовов** — после ~18 вызовов любой следующий получает
  сигнал остановиться.

Цель — не дать модели зациклиться и не переполнить контекст.

---

## Почему выдача инструментов компактная

У GigaChat запрос **с функциями** ограничен ~4096 токенами, а каждый
шаг стадии 1 несёт все 11 схем инструментов.

Поэтому инструменты экономят токены:

- отдают только нужные поля строки, пустые — убирают;
- числа округляются, лимиты строк низкие;
- `get_metric` / `compare` по умолчанию берут агрегат.

Стадия 2 функций не несёт — там доступен полный контекст модели.

---

## Путь вопроса через инструменты

Вопрос: «Почему у Иванова упала метрика X?»

```
1. describe_metric(X)          тип и смысл метрики
2. find_flags(kind='trend')    подтвердить спад
3. metric_tree(X, Иванов)      какая часть просела
           │
           ▼  собранные данные
     Стадия 2 — синтез ответа
```

Каждый факт — отдельный вызов инструмента; модель ничего не выдумывает.

---

## Итог

- Агент работает **только инструментами** — 11 типизированных,
  без сырого SQL.
- Каждый закрывает свою задачу: навигация, разрешение имён,
  получение метрик, готовые выводы.
- Две стадии: инструменты собирают факты → отдельный вызов
  синтезирует ответ.
- Ограничители цикла и компактная выдача удерживают сбор в рамках
  контекста модели.
